# SPEC — MCP Custom Standard

## 1. Objective

Build a reusable **Custom MCP Server Standard** allowing business-process modules to be plugged in per domain: HR, Sales, Finance, Operations, etc.

The server must expose MCP tools to AI chatbots and route each tool execution to deterministic backend services, APIs, or workflows.


**Client onboarding / deployment flow:**

New client
→ Deploy / configure the MCP Custom Standard
→ Set up the company: users, roles, permissions, Google Drive/Sheets/Docs/Gmail
→ Enable the necessary business modules: HR, Sales, etc.
→ Define business processes: recruitment, training, sales pipeline, etc.
→ Add/configure business tools
→ Test with Claude / ChatGPT-compatible MCP client
→ Client starts using the system.

**Example in HR:**

V1 should start with one modular HR MCP server:

```
HR MCP
→ recruitment
→ training
→ performance
→ HR self-service
→ HR knowledge base
```

If isolation becomes necessary, the HR MCP can later be split by exposure level:

```
MCP HR Self-Service
→ accessible to employees and managers
→ HR enquiries, recruitment requests, training requests

MCP HR Operations
→ accessible to HR and HR Directors
→ recruitment processing, performance management, approvals, sensitive reporting
```

## 6. Business Process Runtime Requirements

The MCP standard must support business processes composed of multiple steps, human validations, side effects, and persistent statuses.

This layer is required to plug domain processes such as HR recruitment, sales pipeline, finance approval, onboarding, or support workflows.

### 6.1 Process Model

Each business process must define:

```yaml
process_id: stable_process_identifier
domain: hr | sales | finance | operations | other
name: human_readable_process_name
version: semantic_version
steps: ordered_business_steps
statuses: allowed_status_values
roles: allowed_user_roles
audit_required: true
```

A process is not equal to a single MCP tool.

Rules:

```text
One process = one coherent business workflow.
One MCP tool = one stable business action inside or across the process.
Human decision steps must remain explicit statuses, not hidden AI actions.
All side-effect steps must be auditable.
```

### 6.2 State and Status Management

The standard must support persistent process state.

Minimum state fields:

```yaml
process_instance_id: string
process_id: string
company_id: string
current_status: string
current_step: string
created_by: string
created_at: datetime
updated_at: datetime
last_tool_called: string
external_references: object
audit_log_id: string
```

State can be stored in an external system such as Google Sheets, database, or business application.

The MCP standard must not hardcode the storage backend.

### 6.3 Human Validation Checkpoints

The standard must support explicit human validation checkpoints.

Examples:

```text
manager_validates_job_description
hr_validates_distribution_channels
hr_validates_candidate_analysis
manager_validates_interview_notes
finance_validates_expense_request
sales_validates_proposal_before_send
```

Rules:

```text
AI can prepare recommendations.
AI must not silently approve human decision steps.
Validation status must be stored.
Validation actor must be logged.
Validation timestamp must be logged.
```

### 6.4 Process-Aware Tool Design

Tools must be designed around stable business actions, not raw technical steps.

Example:

```text
Good:
submit_job_request
generate_distribution_kit
analyze_candidate_application
prepare_interview_brief
update_candidate_status

Bad:
write_row
send_http_request
create_file
parse_text
call_google_api
```

Each process-aware tool must declare:

```yaml
tool_name: string
process_id: string
allowed_statuses_before_execution: string[]
status_after_success: string
required_role: string
side_effects:
  - create_document
  - update_sheet
  - send_email
  - create_calendar_event
audit_level: none | standard | strict
```

### 6.5 Google Workspace Process Connectors

The standard must include connector abstractions for Google Workspace process execution.

Minimum connector capabilities:

```text
Google Drive: read files, search folders, store documents
Google Docs: create documents from templates, update placeholders
Google Sheets: read rows, append rows, update statuses
Google Forms: create or reference forms, collect application data
Gmail: create drafts, send validated emails
Google Calendar: create events, update events
```

Connectors must be called through services, never directly from tool handlers.

### 6.6 Audit Log Requirements

Every process action must write an audit event.

Minimum audit event fields:

```yaml
audit_log_id: string
timestamp: datetime
company_id: string
process_id: string
process_instance_id: string
tool_name: string
actor_role: string
actor_identifier: string
input_summary: object
external_outputs: object
status_before: string
status_after: string
result: success | error
error_code: string | null
```

The audit log must support later troubleshooting, compliance review, and process reconstruction.

### 6.7 Module Requirements for Business Processes

A module that implements a business process must include:

```text

├─ process.config.ts
├─ process.statuses.ts
├─ tools/
├─ schemas/
├─ services/

├─ audit/

Required exports:

```

### 6.8 Acceptance Criteria

The implementation is complete when:

```text

```

## 8. Tool Contract

Each MCP tool must define:

```yaml
name: stable_machine_readable_name
description: short_action_oriented_description
input_schema: strict_json_schema
permission_scope: required_business_permission
handler: deterministic_service_function
returns:
  status: success | error
  data: object
  audit: object
```

Rules:

```text
One tool = one business action.
Avoid technical tools exposed to the chatbot.
Prefer coarse-grained business tools over many micro-tools.
All inputs must be validated before execution.
All executions must be logged.
All side-effect actions must return traceable IDs.
```

## 12. Security Requirements

```text
Reject unauthenticated requests.
Check permissions before tool execution.
Validate all inputs.
Do not expose secrets in logs.
Store secrets in environment variables.
Support per-company config.
Support tool-level permission scopes.
Log all side-effect actions.
Return safe errors to chatbot.
Keep internal stack traces server-side only.
```

## 14. Standard Upgrade Policy

The core standard must be versioned independently from business modules.

```text
core version: semantic versioning
module version: semantic versioning
breaking changes: forbidden without migration notes
deprecated functions: keep for one minor version
```

Add a `STANDARD_CHANGELOG.md` file.

Each upgrade must verify:

```text
MCP server still starts
All tools are registered
All schemas are valid
All module tests pass
All connector health checks pass
Docker image builds
```
